# 接私活指南

> 一句话：大小通吃，万物皆可接——从 1 万的小页面到 50 万的全栈项目，明确边界、守住底线、持续积累。

## 定位

### 对外一句话

> **独立全栈开发者 · 5 年企业级开发经验**
> 
> Web 全栈 / 数据可视化 / 系统架构 / 政务数字化
> 
> 提供从技术方案到落地交付的全流程服务。

### 内部原则（对自己用的，不写在对外材料上）

- **项目灵活** — 从几千的企业官网到几十万的全栈系统，根据需求评估，不设门槛
- **价格透明** — 按工作量合理报价，不虚高也不低价竞争
- **质量一致** — 无论项目大小，交付标准不降级
- **现金流安全** — 定金 + 阶段款 + 尾款分步走，不垫资

- 技术方案评估
- 代码审查
- 部署运维
- 已有项目改造/扩展

## 定价策略

### 核心原则

- **网站上不标价格** — 价格放网站上 = 把自己当商品。客户问的时候再根据需求评估。
- **先评估后报价** — 了解需求复杂度、周期、维护成本后再定价，不在不清楚情况时报价。
- **报价锚点抬高再让步** — 第一次报价给合理偏上的价格，如果客户预算不够再调整范围，而不是直接降价。

### 内部定价公式

```
报价 = 预估工时（天） × 日薪 × 1.5（风险溢价）
```

日薪参考：
- 简单页面/模板套改 → ¥1,000/天
- 正常项目开发 → ¥1,500-2,000/天
- 紧急/复杂项目 → ¥2,500-3,500/天

## 收款流程（守底线）

```
定金 30%-50% → 第一阶段交付验收 → 第二阶段款 30% → 验收 → 尾款 20%-40%
```

- **1 万以下项目**：定金 50%，验收后 50%
- **1-5 万项目**：定金 30% + 中期 40% + 尾款 30%
- **5 万以上项目**：按里程碑分期付款，每阶段验收后付款

**打死不做的**：先干活后付全款、无限改需求不加钱、不给定金。

## 获客渠道

### 第一梯队（你现在就能用）
1. **朋友圈/熟人介绍** — 发一条"开放接单，大小都接"，这是最稳定的来源
2. **已有行业群** — 政务信息化群、IT 交流群，偶尔冒泡展示作品
3. **闲鱼/猪八戒** — 虽然卷但有时候能捡漏，价格别太低

### 第二梯队（中期建设）
4. **个人网站 + SEO** — 你的网站就是最好的名片
5. **技术文章** — 知乎/掘金发项目复盘文章，"接私活"本身就是流量词
6. **GitHub 持续更新** — 项目代码放上去，就是技术能力的活简历

### 第三梯队（长期）
7. **Upwork / 远程工作平台** — 接海外项目，时薪更高
8. **口碑裂变** — 做好一个客户，他会带来 3 个

## 避坑指南

| 坑 | 怎么避 |
|----|--------|
| 需求不明确就报价 | 先出需求文档/原型，确认了再报价 |
| 客户反复改需求 | 合同里写清楚"超出范围的部分额外计费" |
| 拖欠尾款 | 验收前不交源代码/不部署上线，测试环境验收 |
| 白嫖方案 | 不给详细设计文档，只给演示和原型 |
| 技术栈被客户指定 | 用你熟悉的做，不熟悉的加价学 |

## 随项目积累 — 报价档案

**每个项目必须记录**以下信息，方便后续报价时参考历史数据：

### 项目报价记录表

| 项目名 | 报价 | 实收 | 周期 | 工时（人天） | 折合日薪 | 获客渠道 | 备注 |
|--------|------|------|------|-------------|----------|----------|------|
| [[装卸通]] | ¥159,000 | — | 待定 | — | — | 交通运输局委托 | 建设中，按功能模块报价 |
| | | | | | | | |

### 记录什么

每完成一个项目，补一行：

```
项目名 | 报价 | 实收（到账多少） | 周期（开工→验收） | 投入人天 | 折合日薪 | 客户来源 | 踩了什么坑
```

### 这些数据的用途

- **报价参考** — 同类项目直接翻历史报价，不用每次从零算
- **定价校准** — 记录多了就知道自己实际日薪是多少，报价越来越准
- **获客分析** — 哪个渠道来的客户最多、客单价最高，之后发力
- **避坑备忘** — 每个项目踩的坑记下来，下次同样问题不犯第二次
