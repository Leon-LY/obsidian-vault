## 第一章：项目概述

### 1.1 项目背景

**创始人即用户。** Leon，一对龙凤胎（4 岁，2022 年 7 月 7 日出生）的父亲，在亲身养育双胞胎的过程中发现：市面上所有育儿 App 都是"单胎思维"——记录只能一个宝宝一个宝宝切、生长曲线无法对比、作息管理没有同步协调。市场上唯一的双胞胎专属产品 Twiniversity 仅有英文版且功能简陋。

**中文市场完全空白。** 每年中国新增双胞胎约 15~18 万对（自然+试管），存量 0-6 岁家庭约 80~100 万对。这群父母的育儿难度是单胎的 2 倍以上，但数字化工具的供给几乎为零。

**AI 技术已成熟。** 2026 年，国产大模型（DeepSeek、腾讯混元）已可低成本集成，微信小程序生态完善，云开发技术降低创业门槛。现在是切入这一空白市场的最佳窗口期。

### 1.2 产品定位

> **并蒂星球——0-6 岁双胞胎家庭的 AI 育儿操作系统。0-3 岁解决同步管理，3-6 岁解决差异化陪伴。**

| 维度 | 定位 |
|------|------|
| 目标用户 | **0-6 岁**双胞胎/多胞胎父母（**妈妈 + 宝爸，平等对待**） |
| 扩展用户 | 隔代照护者（奶奶/外婆）、育儿嫂/月嫂、幼儿园老师、**🆕 双胞胎孩子本人（作为产品设计原点）** |
| 服务场景 | 喂养记录、睡眠管理、成长追踪、作息同步、家庭协作、就诊准备、**🆕 分班决策、幼儿园家园共育、差异化发展** |
| 差异化 | 一屏双轨不切换、萌芽日记（关系叙事）、宝爸专属三功能、就诊速查卡、异常事件日志、奶奶无障碍标准、儿科医生安全红线、**🆕 入园助手(分班决策)、双宝能力观察(独立时间线)、V3 同色系无性别暗示** |
| 产品形态 | 微信小程序（MVP）→ 独立 App（Phase 3）

### 1.3 项目愿景

**两阶段叙事**：并蒂星球的产品逻辑随孩子年龄变化。

- **0-3 岁阶段**：做「记录工具」——高效、可靠、免思考。让深夜喂奶的妈妈单手两下就能记完两个宝宝的喂养数据，让加班的爸爸扫一眼快照就知道家里发生了什么。
- **3-6 岁阶段**：做「发现工具」——帮家长看见两个独立个体。入园助手（分班决策）、双宝能力观察（独立时间线）、萌芽日记社交版——不再问"两个宝宝同步了吗"，而是问"两个宝宝各自成为了谁"。

**长期愿景**：从"中国首款双胞胎育儿工具"出发，逐步成长为**双胞胎家庭的数字化操作系统**——覆盖记录管理、知识服务、萌芽关系叙事、宝爸赋能、隔代友好、就诊辅助、家园共育、同城社交、闲置交易、B2B 机构服务。以双胞胎为切入口，横向拓展至多孩家庭管理平台。

> 🆕 V4.0 金句：**"并蒂星球记录的是'一个妈妈带两个娃'；并蒂星球回答的是'两个独立生命如何共用同一个根'。"**

### 1.4 **产品灵魂：看见每一个人**

> 并蒂星球不是一个"记录两个宝宝的 App"。它是中国第一个认真对待以下 **11 种人**的产品：疲惫的妈妈、想参与但不知从哪开始的爸爸、怕弄坏 App 的奶奶、需要一张速查卡的儿科医生、想告诉家长"你的孩子在学校不一样"的老师、想保护自己的育儿嫂、🆕 3 岁时开始问"为什么我和妹妹不一样"的那个孩子——以及那两个正在一起长大的宝宝。
>
> **任何一个被现有育儿 App 忽略的角色，在并蒂星球里都被看见了。这就是护城河。** 功能可以被抄袭，但"看见每一个人"的产品哲学，永远抄不走。

### 🔬 本轮最大发现：文档-代码断裂

> 这是前六轮论证从未触及的维度。技术架构师实际克隆代码仓库后交叉验证发现：

| 方案声称 | 代码实际 |
|------|------|
| "V4.1 新增大便追踪器" | ❌ 无 diapers 相关表/字段/Store/页面 |
| "V4.1 新增爸爸'不知道'按钮" | ❌ 零代码实现，连功能描述都没有 |
| "V4.1 补充服务器灾备" | ❌ 无 pg_dump cron / 备份脚本 / 流复制 |
| "V4.1 补充 CI/CD" | ❌ 无 GitHub Actions / deploy script |
| "建立 DAL 层" | ❌ routes 直写 db.query，无中间抽象 |

## 第三章：产品方案

### 3.1 产品形态策略

```
Phase 1（10周MVP）    Phase 2（3个月）      Phase 3（6个月后）
微信小程序             小程序 + Web管理端     独立App (iOS/Android)
──────────             ──────────────        ────────────────
双轨记录+萌芽日记        AI作息引擎             离线模式
对比生长曲线+宝爸功能    AI育儿顾问             Push精准推送
双胞胎知识库+守护系统    会员付费体系            IoT硬件联动
新手村引导+传播引擎      社区上线                B2B机构版
种子用户验证PMF         同步率PK赛
```

**选择小程序 MVP 的理由：**
- 宝妈微信渗透率 90%+，微信群是核心传播阵地
- 零安装摩擦（扫码/转发卡片即用）
- 微信支付原生支持，付费路径短
- 双胞胎对比图自带社交裂变基因

**后续必须做 App 的理由：**
- 小程序无法做自由 Push（喂养/疫苗提醒是留存核心）
- 儿童隐私数据需要完全本地化（App 比小程序更安全）
- 未来 IoT 硬件（智能体温贴/睡眠监测带）必须 App

### 4.4 核心技术决策

#### 决策 1：在 MVP 就建立数据访问层（DAL）

**原因**：如果云函数直接写 SQL，到了 DAU 增长期想加缓存、切读写分离、加物化视图——需要改所有云函数。

**方案**：创建 `services/` 目录，云函数只调 service 接口，service 内部封装查询逻辑、缓存策略、权限校验。

#### 决策 2：关键云函数上预置并发

| 函数 | 预置实例 | 月费 |
|------|:---:|:---:|
| 首页数据 | 1 | ~¥30 |
| 宝宝列表 | 1 | ~¥30 |
| 生长曲线 | 1 | ~¥30 |

> 冷启动延迟 1-2s，预置后降至 <100ms。对用户体验影响巨大。

#### 决策 3：ECharts 分包加载

避免主包超过 2MB 限制。在 `pages/growth/` 分包中引入 ECharts，主包不包含图表库。

#### 决策 5：🆕 V4.0 ECharts 按需懒加载

**实际情况**：npm ECharts 被 uni-app Vite 打包进 `common/vendor.js`（1.15MB），导致小程序启动超时。已修复：ECharts 放入 `static/echarts/echarts.min.js`（500KB），`ec-canvas.vue` 用 `require.async` 按需加载。`vendor.js` 从 1.15MB → 71KB（↓94%）。方案中标注此决策，避免后续开发者误回退。

#### 决策 6：🆕 V4.0 本地持久化层

**当前**：使用 `utils/persist.ts`（`uni.setStorageSync` + `tp_` 前缀）做 Pinia Store 本地持久化。启动时 hydrate 恢复状态。**迁移路径**：后端上线后，persist.ts 的抽象层只需替换底层实现为 `fetch` API，Store 层零改动。

#### 决策 7：🆕 后端优先级排序

P0: 数据库建表 + `wx.login` → JWT 鉴权 → P1: Store 本地持久化 → HTTP API 迁移 → Docker 部署 → P2: Redis 会话 + 微信订阅消息 + DeepSeek API 代理

- 前端缓存 LMS 参数作为静态 JSON
- z-score 和百分位在前端计算（公式简单，无需后端）
- 后端只负责存储和查询测量记录

### 4.5 安全与合规（含儿科医生安全红线 🆕）

#### 通用安全

| 维度 | 措施 |
|------|------|
| **传输加密** | HTTPS + TLS 1.3 |
| **存储加密** | COS 服务端加密 SSE-COS、AES-256-GCM 加密手机号 |
| **数据最小化** | 照片只存缩略图（256×256），原图 24h 删除 |  
| **数据脱敏** | 宝宝昵称脱敏存储；DeepSeek API 调用前替换为"宝宝A/宝宝B" |
| **数据删除** | 用户注销后 30 天内物理删除所有关联数据 |

#### 儿科医生安全合规红线 🆕

> 以下 6 条由儿科医生独立审计提出，**每一个功能上线前必须逐条对表检查**。

| # | 红线 | 具体规则 |
|:--:|------|------|
| 1 | **不做医疗判断** | 任何功能不得输出"建议就医""建议观察""可能是什么病" |
| 2 | **不做疫苗决策** | 疫苗冲突检测只呈现"近 7 天有过发烧记录"，不判断能不能打 |
| 3 | **不做症状分析** | 异常事件日志是纯文本便签，不做任何 AI 分析、不分类、不关联 |
| 4 | **颜色=统计，≠诊断** | 就诊速查卡的差异率颜色标注须附声明："颜色仅为统计标记，不代表医学诊断" |
| 5 | **所有 AI 内容须标注免责** | "本工具提供的信息仅供参考，不构成医疗建议" |
| 6 | **照片加密存储** | 异常事件日志拍摄的照片（皮疹、步态等）属于高度敏感健康数据，COS SSE-COS 加密，仅存缩略图 |
