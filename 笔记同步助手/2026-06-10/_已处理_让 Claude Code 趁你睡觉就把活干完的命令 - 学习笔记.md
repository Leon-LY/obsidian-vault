## 核心思路：一次说清楚，放手让它自己干

使用 Claude Code 时，新手容易陷入“手动循环”：每次改完看一眼，觉得不够再补一句，人反而比 AI 更累。真正省事的做法是：把目标一次性写清楚，然后让 Claude Code 自行迭代，直到完成。关键在于不依赖 auto mode 或更长提示词，而是先定义“完成”的标准。下文逐一讲解几个命令及其解决的核心卡点。

## 1\. Goal —— 设定完成条件，自动判断是否终止

### 解决的问题

每轮都需要用户手动推进。只要给出明确的完成条件（如：某组测试全部通过、登录页正常渲染、验收清单全部满足），Claude Code 会在每轮结束后用小模型评估目标是否达成，不会再停下来等你下一条指令。

### 使用要点

-   条件不能写得太虚。比如“修好 bug”太模糊，需具体到“test.ts 里的测试全部通过，npm test 退出码为 0，且不修改其他测试文件”。
-   Goal 的条件最多可写 4000 字符，可以写细，还可以写安全边界（如最多跑 20 轮或 30 分钟后停止）。
-   `goal` 不带参数查看状态，`goal clear` 取消当前目标。

## 2\. 配合 Auto Mode 实现真正自主运行

很多人误以为开了 auto mode 就能全程自动。实际上，auto mode 只管单轮内的工具确认（减少确认次数），不会自动开启下一轮。完整的自主流程是：

1.  写一份清晰的 spec，包含目标、验收标准、不能碰的边界；
2.  清空上下文；
3.  用 `goal` 让 Claude Code 按 spec 迭代，直到测试跑完、验收结果交代清楚；
4.  再开 auto mode 减少中途工具确认。

这样你就可以走开，甚至用 remote control 在手机上查看。若要在 CI 里跑，可用 `claude -p` 带上 goal 做非交互执行。前提是 workspace 已被信任，hooks 未被策略禁用。

## 3\. Loop —— 按计划轮询等待外部状态变化

### 解决的问题

不是判断任务做没做完，而是定期查看外部状态是否变化。例如：部署还在跑、长构建未结束、集成测试等结果。适合场景：不是让 Claude Code 一直推代码，而是按时间表反复查看状态。

### 使用方式

-   可以写固定间隔：`loop 五分钟检查部署状态`。
-   也可以不给固定间隔，让 Claude Code 根据上次结果自己决定等待时长（结束快则等短点，没动静则等久点）。
-   可以不给 prompt，让它照着 `claude dir/loop 配置文件` 里的默认指令来。

注意：loop 适合在当前会话里轮询，有会话和过期边界。真正的被动事件（如 CI 挂了、代码评审有留言）更适合用事件推送，而不是轮询。

## 4\. Batch —— 大任务拆分与并行执行

### 解决的问题

将一句改动描述自动拆成 5-30 个相对独立的单元，每个单元交给一个后台 sub-agent 在自己的工作树里实现和测试，最后整理成独立改动。适合跨代码库的大活，如框架迁移、批量模块改造。

### 风险与要点

-   核心风险不在执行，而在拆分。拆分得好每块改动清楚，拆得不好会变成一堆互相打架的分支。
-   最不能偷懒的是认真审查它拆出来的计划：边界、依赖、文件范围。没看明白千万别急着批准。

## 5\. Simplify —— 改动后的代码清理与优化

### 解决的问题

不是写新功能，而是在你或 Claude Code 做完改动后，让它回头看：有没有现成 helper 可复用？代码能否更简单？效率有无坑？抽象层级是否适当？它会并行跑几个 review agent，汇总建议并应用安全修复。

### 版本差异

从 v2.1.154 起，simplify 不再主要查 correctness bug，此类检查应使用 `code review`。更早版本中它更建议用 `code review fix`。最佳用法：将 simplify 当作提交前的清理工，真正的逻辑判断靠测试、类型检查、lint、code review，最后人工审核。

## 6\. Debug —— 排查 Claude Code 自身的问题

### 解决的问题

不是业务代码问题，而是 Claude Code 自己出毛病：hook 该触发却没触发、skill 没被调用、MCP server 连上了但工具没显示、一直建议弃用 API 等。直接跑 `debug` 比手动翻日志快得多。

### 使用方式

-   它会为当前 session 打开 debug log，读日志帮你判断问题。
-   可以带一句提示让它聚焦某个现象。
-   配套 `doctor`：如果是安装/环境配置层面的问题，先跑 doctor；能跑但时灵时不灵再跑 debug。

简单判断：完全跑不起来先看 doctor；能跑但时灵时不灵再看 debug。

## 7\. 命令主线总结

| 命令 | 用途 |
| --- | --- |
| goal | 让 Claude Code 围着完成条件自动迭代 |
| auto mode | 减少单轮内的工具确认次数 |
| loop | 按计划轮询等待外部状态变化 |
| batch | 大任务拆成多块让后台代理并行处理 |
| simplify | 改动后的代码清理和优化 |
| debug / doctor | 诊断工具链自身问题 |

自主不等于完全不管。Claude Code 能自主执行的前提是你先定义清楚终点。如果 spec 清晰、验证手段足够硬，它确实可以在你睡觉时做完一块功能；但如果 spec 错误、验证松垮，它也会高效地产出方向错误的代码，且测试全过。

## 补充说明

-   写 spec 时务必具体，避免模糊目标。
-   使用 goal 前确保 workspace 受信任，hooks 未被禁用。
-   loop 适用于轮询，但对实时事件建议改用事件推送。
-   batch 一定要仔细审查拆解计划。
-   simplify 在新版本中不再查 bug，需配合 code review。
-   debug/doctor 用于诊断 Claude Code 自身问题，按“先 doctor 后 debug”顺序。
